Troubleshooting Guide

Major incident — database down or degraded

1.First response (incident management)

  • Record who raised the problem and the time it was reported.
  • Restoration of service — what is the agreed timebox? How long may diagnosis run before service must be restored (e.g. by failover or DR)?
  • Start a recurring 15-minute timer.
  • Notify relevant parties of the problem, and how frequently you will provide updates.
  • If it is a serious problem, verify who is taking charge — is ECAB required?
  • Record any pertinent findings at every step.
  • If a possible cause is identified, verify or dismiss it ASAP.
  • Record progress every 15 minutes and notify relevant parties.
Restore service first — but safely. Before failing over to restore service, establish whether the fault is local (a failover resolves it) or systemic / a shared dependency such as disk, storage fabric, network or AD (a failover only relocates it). Failing over into an unresolved shared fault doesn’t restore service — it reproduces the outage on another node and spends your best recovery option.

2.Triage — is the cause obvious?

  • Is the cause obvious? E.g. disk full, blocking, locked AD account. Is it a problem that could be fixed by a failover — i.e. is it local?
    • If so, communicate it, confirm the safe remediation (fix in place vs. failover, per the local-vs-systemic test above), and have remediation ready.
  • If the cause is not obvious, work the diagnostic sweep below.

3.Diagnostic sweep (cause not obvious)

Connectivity

  • Are new connections possible? Can you connect? Are others able to connect? Is there a noticeable delay in connecting?
  • NOTE: if you can’t connect, use the DAC (Dedicated Administrator Connection).

Is anything in the instance broken?

  • Agent jobs — check for failures.
  • Logins — AD account dropped, or password changed.
  • Databases — data file or transaction log full; Service Broker not working; Compatibility Level.
  • SSIS — check for issues.

Any recent changes?

  • Code — have someone check change control, and ask around.
  • Data (bulk load, etc.).
  • Host, instance or database.
  • Agent jobs (function & schedule).
  • Recently installed applications.
  • Check whether any recently departed employee still has an active login (Active Directory & SQL Server).

Infrastructure

  • Ask the relevant people to check network, servers, shared storage, local storage (e.g. the C: drive filling up), antivirus, DoS, etc. — while you continue checking the SQL side.

Patching

  • Is any taking place right now?

Data centre

  • Any emails from the provider about outages? Any way to check independently?
Timebox reached and still down? Escalate — and if the fault is local, restore service by failover / DR now and continue diagnosis in parallel. Don’t let “just checking” run past the agreed window while the business is down.

4.Deeper SQL diagnostics

  • Run sp_WhoIsActive & sp_BlitzFirst, with outputs to tables, and compare to normal values. (You should be keeping a history of these outputs anyway.)
    • sp_WhoIsActive: look for blocking, deadlocking, or an unusually high number of sessions.
    • sp_BlitzFirst: check the top 5 waits; flag any with unusually high values, or that aren’t usually in the top 5.
  • Check Perfmon counters, with output to a table, and compare to normal values. (Keep a history of these anyway.)
    • Physical Disk — Avg. Disk sec/Read
    • Physical Disk — Avg. Disk sec/Write
    • Physical Disk — Disk Reads/sec
    • Physical Disk — Disk Writes/sec
    • Processor — % Processor Time
    • SQLServer: General Statistics — User Connections
    • SQLServer: Memory Manager — Memory Grants Pending
    • SQLServer: SQL Statistics — Batch Requests/sec
    • SQLServer: SQL Statistics — Compilations/sec
    • SQLServer: SQL Statistics — Recompilations/sec
    • System — Processor Queue Length
  • Check Query Store — look for evidence of parameter sniffing and correlate with sp_WhoIsActive output. Force a good plan if possible.
  • Find the most expensive queries and, if relevant, back-trace them to stored procedures (sp_BlitzCache).
  • Check the SQL Server log & the Windows Event Log (use PowerShell: https://bit.ly/3YdBiyg).

5.After any fix / once stable

  • DR/HA — check it’s working after any fixes are applied.
  • Check the SQL Server instance reports.
  • Confirm service is restored and stable, then close the loop: hold a post-incident review and raise a problem record so the root cause is fixed permanently — not just worked around.
